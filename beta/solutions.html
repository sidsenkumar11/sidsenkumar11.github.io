Meeting 8 Solutions

Beginner - RSA Introduction

We know that the public key consists of e and n. The private key consists of d, the decryption exponent.
In this problem, we are actually given d, the decryption key, so we have nothing to really do but decrypt it.
So, to convert the given ciphertext to plaintext, simply take c, raise it to the d'th power, and mod it by n.

Intermediate - Factorization

The given n is a pretty small number for a computer. Plugging it into wolfram alpha and asking for a prime factorization, we can see that it factors into two distinct primes, our p and q.
Using the given p and q, we can calculate totient(n) as (p-1) * (q-1).
Use totient(n) to find the decryption exponent d, calculated as e*d = 1 mod totient(n).
Once we have the decryption exponent d, simply do c^d mod n to find the plaintext p.

Intermediate - Small Private Key and Known Plaintext RSA

This time, we are given an n and a ciphertext c. Note that n is relatively small and easy to factor.
We can again use wolfram alpha to factor into its p and q. Now, we know p and q so we can calculate totient(n) as (p-1) * (q-1).
Now, normally we would use totient(n) and e to calculate the decryption exponent d, using the formula e*d = 1 mod totient(n). However, we are not given an encryption exponent e in this problem! So what do we do?
Use the fact that the given ciphertext is known to contain the text "rsa", and brute force encryption exponents that are co-prime with n until a resulting ciphertext yields plaintext that contains the text "rsa".

Beginner - Simple SQL Injection

First, let's take a look at standard SQL syntax. SQL deals with databases that contain tables.
Traditionally, a statement to retrieve information from such a table follows as:
SELECT * FROM users WHERE name = 'userName'
SELECT, FROM, and WHERE are keywords in SQL.
* means "all things".
users is the name of this table
name is a column in the table.

So, the SQL statement SELECT * FROM users WHERE name = 'admin' will return the line in the table where the username is admin. Of course, the solution to this problem isn't that simple - the author of the problem chose to write code that prevents the admin password from being directly accessed.

Typically, the way the system will call this SQL statement is by passing whatever you input as your username and password in those fields into the query.
So, a query would look something like: select * from users where username='$username' and password='$password'
where $username is a string of text and $password is a string of text that the user provides to the website in a textfield.
Now, how can we use SQL injection to give us things that are not meant to be sent to us?

With our vulnerable website, we only have one field - username. So likely, the SQL statement will be something like:
select * from users where username='$username'
To retrieve all the table data, we will simply set:
$username = ' or '1'='1
The new query will become:
select * from users where username='' or '1'='1'
(If that doesn't make sense, just replace $username in the original query with the text that I set $username to).
This statement basically says "Give me every line in the table where the username is '' OR where '1'='1'". Since '1'='1' is true no matter which row of the table you are in, the output wil be every row in the table.


We can see that is exactly what happens when we enter in this input. Scroll to the bottom of the page and get the admin password.

Flag: bad_code_and_databases_is_no_fun

Intermediate/Expert - Hidden Site

Note: You could brute force the URL for this problem or go to my Github commit history to find the solution, but as the intention was to get you familiar with Burp Suite, this is not the recommended solution.

First, as the hint suggests, we are going to download Burp Suite. Burp Suite is a jar file that should be in the toolkit of any network CTF problem solver. What is Burp Suite? It is a program that analyzes all network traffic coming into and out of your browser. This is very useful for finding things that may not directly be visible on the actual rendered page. You can find the link here.

Next, we need to configure our browser to send network traffic through Burp Suite. This is easily done on browsers like Firefox by opening the Advanced tab and changing the proxy settings. You can find a tutorial on how to do this here.

Finally, open up Burp Suite (can be run using java -jar) and open up the browser with the configured proxy settings. Note that the problem hint says "Spider". This is a function in Burp Suite that essentially crawls the pages you visit on the browser and makes a map of all links, images, and pages it finds. Go to the "Spider" tab in Burp Suite and click "Spider is paused" so that Spider will be enabled. Now, let's visit the url in question (i.e. sidsenkumar11.github.io).

Browse around the website and click on as many links and pages as possible. (If the browser seems to be stuck on "Connecting", see Note 2 below). If you click on some of the project links, like "Password Manager", you will reach in-progress.html. If you've done this, go back to Burp Suite.

On the left pane of Burp Suite, expand the arrow next to sidsenkumar11.github.io to see what Spider has collected information on.
Notice how there is a page it found called "theflag.html"? This seems suspicious! Let's visit sidsenkumar11.github.io/theflag.html
Aha! The page renders and there is text on it! The word "INTERMEDIATE" should tell you that we are close to the flag since this is the Intermediate problem. Beginner - Bad Hash Algorithm

Take a look at the hashing algorithm to see what is happening to see if we can reverse engineer a ciphertext.
For every character in the plaintext:

1) ord(c) gets the ASCII value of the character. Ex. 'A' becomes 65.

2) This number has an operation done to it; namely //key. We know key to be 3. A google search reveals that the "//" operation in python is floor division (integer division). For example, this means that 81//3 = 82//3 = 83//3 = 27.

Let's now look at the first number in the ciphertext: 27.
27 is the quotient after the ascii value of the first letter in the plaintext was floor divided by 3. What are the possible numbers that can result in 27?

Well, as I showed in 2), all three of 81, 82, and 83 yield 27 when we floor divide by 3!
So, for 27, we know that the plaintext ascii value is either 81, 82, or 83. These correspond to the ASCII characters 'Q', 'R', and 'S' (you can use an ASCII table to verify this).
